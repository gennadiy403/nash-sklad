# OborotCRM (Оборот)

## What This Is

Бизнес-CRM для продавцов: учёт, аналитика, прогноз и команда в одном месте. Точка входа — продавцы маркетплейсов (WB, Ozon, Яндекс.Маркет); масштаб — любой торговый бизнес (розница, опт, интернет-магазины). Существующий продукт МойСклад накопил 8 лет UX-долга и неработающие интеграции через прослойки (TopSeller) — OborotCRM строится как прямая замена с нативными интеграциями, модульной архитектурой и бесконечным хранением истории аналитики.

Продукт делает один человек — основатель SKVO Clothes (действующий продавец на МП в России и Казахстане, веб-разработчик 12 лет) — плюс небольшая команда энтузиастов. Реализатор для GSD-фаз — Claude.

## Core Value

**Продавец, живущий на маркетплейсах, может вести учёт и смотреть аналитику без прослоек и без потери данных — нативные интеграции WB/Ozon/ЯМ + бесконечное хранение истории.**

Если всё остальное сломается, это должно работать.

## Context

**Основатель и команда:**
- Основатель — действующий продавец (бренд одежды SKVO Clothes, WB/Ozon/ЯМ + сайт + шоурумы), 8 лет на МойСклад, 12 лет в вебе, работает в России и Казахстане.
- Небольшая команда энтузиастов + Claude как реализатор GSD-фаз.
- Продукт "сделан для себя" — аутентичность важнее быстрого вывода на рынок.

**Технический контекст (из .planning/codebase/):**
- Монорепо npm workspaces: `apps/landing/` (Vite + React Router, живой) и `apps/app/` (Next.js 14 + Supabase, скелет).
- Текущая интеграция: Google Sheets webhook + Telegram Bot API — обе требуют замены на собственный бэкенд. Telegram-токен в VITE_* (client-side) — security issue, нужно перенести на сервер.
- Supabase клиент инициализирован, но ни auth, ни data-fetching не реализованы.
- Нет тестов, нет CI, нет error tracking.
- Названия пакетов ещё "nashsklad" — нужен ребренд в коде.
- Зависимость framer-motion установлена, но не используется.
- Платформенные риски: Vite preview используется как production-сервер на Railway — не масштабируется.

**Пользовательский контекст:**
- Боль аудитории: МойСклад за 8 лет не изменился, TopSeller глючит, учёт сбивается, лимиты API МП не позволяют восстановить историю. "Смирились с тем что нормальный учёт невозможен".
- Эмоциональные якоря для продукта: "8 лет терпел", "продавец делает для продавцов", "друзья оторвали с руками", "вместе вылизываем".
- Основной сегмент года 1 — Сегмент B (малая команда 2–5 чел, 200–2000 SKU, 2 500–5 000 ₽/мес).

**Правило качества (важно для приоритизации):**
> Продукт важнее маркетинга. Платный маркетинг включаем только когда month-1 retention >70%. Первые 3 месяца — онбординг каждого пользователя вручную (звонок, вопрос, фикс).

## Constraints

- **Tech stack**: React 18 + Vite (landing) / Next.js 14 + Supabase (app) / Tailwind CSS — уже зафиксировано существующим кодом, монорепо на npm workspaces. Менять только по ADR.
- **Data residency**: Россия + Казахстан — основатель оперирует в обеих юрисдикциях; хостинг и Supabase-регион должны учитывать это (EU-регион или собственный self-hosted PG в RU, либо мульти-регион — решается ADR-ом).
- **Infinite analytics retention**: все данные с маркетплейсов хранятся бессрочно с первого дня подключения. Рост хранилища — сознательный trade-off. Это USP и архитектурное ограничение, а не feature — storage-стратегия, партиционирование, cold storage всё учитываем с нуля.
- **Нативные интеграции, без прослоек**: никакого TopSeller, никаких промежуточных SaaS. Прямая работа с API WB/Ozon/ЯМ.
- **Rate limits API МП**: акценты на throttling, retry, очереди, инкрементальную синхронизацию — историю восстановить невозможно, значит синхронизация должна быть надёжной с первого дня.
- **Early Bird навсегда**: первые 500 платящих получают 50% скидку пожизненно — биллинг должен уметь cohort-based pricing, разные цены для разных users на одном и том же тарифе.
- **Product-first gate**: платный маркетинг off, пока month-1 retention <70%. Не торопимся выкатывать публично.
- **Соло-разработчик + Claude**: нет команды для сложных процессов; предпочитаем простые решения и монолит там, где микросервисы не оправданы.
- **Бюджет на разработку (не на маркетинг)**: не обозначен явно, но подразумевается "как можно дешевле". Избегать дорогих managed-сервисов, где Supabase + Railway закрывают потребность.
